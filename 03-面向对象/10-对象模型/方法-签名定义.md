### 1. 先明确：方法签名的「官方定义」和「通俗理解」

#### 官方定义

在 Java 中，**方法签名（Method Signature）= 方法名 + 参数列表（参数的类型、个数、顺序）**。

注意：**方法的返回值类型、访问修饰符（`public`/`private`等）、`throws`异常声明，都不属于方法签名的一部分**。

#### 通俗理解

可以把方法想象成一个「快递柜」：

- 「方法名」就是快递柜的「编号」（比如 A01）。
- 「参数列表」就是快递柜的「格子规格」（比如能放 1 个大包裹、2 个小包裹，顺序不能乱）。
- 只要「编号」不同，或者「格子规格」不同，就是两个不同的快递柜（不同的方法签名）。
- 而快递柜的「颜色」（返回值类型）、「是否对外公开」（访问修饰符），不影响快递柜的唯一标识。

### 2. 方法签名的核心作用：区分不同方法（尤其是方法重载）

Java 中，同一个类中可以存在多个同名方法（这就是「方法重载」），而区分它们的唯一依据，就是**方法签名不同**。

#### 示例 1：合法的方法重载（签名不同）

下面的`Person`类中，两个`setName`方法是合法重载，因为它们的方法签名不同：

java

运行

```
class Person {
    // 方法1：签名 = setName + (String)
    public void setName(String fullName) {
        // 实现
    }

    // 方法2：签名 = setName + (String, String)（参数个数不同，签名不同）
    public void setName(String givenName, String surname) {
        // 实现
    }

    // 方法3：签名 = setName + (String, int)（参数类型不同，签名不同）
    public void setName(String name, int age) {
        // 实现
    }

    // 方法4：签名 = setName + (int, String)（参数顺序不同，签名不同）
    public void setName(int age, String name) {
        // 实现
    }
}
```

这 4 个方法同名，但参数列表不同（个数、类型、顺序），因此方法签名不同，属于合法的方法重载，编译器能准确区分。

#### 示例 2：非法的「伪重载」（签名相同，编译报错）

下面的写法会编译报错，因为它们的方法签名相同，编译器无法区分：

java

运行

```
class Person {
    // 方法1：签名 = setName + (String)
    public void setName(String name) {
        // 实现
    }

    // 错误1：仅返回值类型不同，签名相同（编译报错）
    public String setName(String name) {
        return name;
    }

    // 错误2：仅访问修饰符不同，签名相同（编译报错）
    private void setName(String name) {
        // 实现
    }
}
```

这两个方法的「方法名 + 参数列表」完全一致，即使返回值类型、访问修饰符不同，方法签名也相同，不属于合法重载，编译器会提示「方法已被定义」。

### 3. 补充：构造方法也有「签名」

和普通方法一样，构造方法也有签名，构造方法的签名 = 构造方法名（即类名） + 参数列表。

这也是为什么一个类中可以有多个构造方法（构造方法重载），比如：

java

运行

```
class Person {
    // 构造1：签名 = Person + ()
    public Person() {}

    // 构造2：签名 = Person + (String, int)（签名不同，合法重载）
    public Person(String name, int age) {}
}
```

### 总结

1. Java 方法签名的核心组成是「方法名 + 参数列表（类型、个数、顺序）」，返回值和访问修饰符不属于签名。
2. 方法签名是区分同名方法的唯一依据，也是方法重载的判断标准。
3. 构造方法同样拥有签名，支持基于参数列表的重载。
